# Style Knowledge Base Schema

# Style Name

Microsoft Technical Writing

## Purpose

Create clear, helpful, accessible, and task-oriented technical documentation.

## Audience

End users, business users, administrators, and technical users.

## Tone

Warm, approachable, supportive, and professional.

## Voice

Use active voice whenever possible.

## Writing Principles

- Be clear and concise.
- Make every word matter.
- Focus on helping users accomplish goals.
- Use plain language.
- Avoid unnecessary jargon.
- Write for a global audience.
- Use inclusive and accessible language.

## Headings

### Heading Rules

- Use descriptive headings.
- Organize content logically.
- Make headings easy to scan.

### Capitalization Rules

- Use sentence case for headings.

## Terminology

### Preferred Terms

- Use Microsoft-approved terminology.
- Use terminology consistently throughout a document.

### Avoided Terms

- Avoid unofficial variants.
- Avoid inconsistent terminology.

### Terminology Examples

- Use "dropdown" instead of "drop-down".
- Use "sign in" as a verb.
  Example: Sign in to your account.

- Use "sign-in" as an adjective.
  Example: Sign-in page.

- Use "pane" instead of "blade".

- Use "allow list" instead of "whitelist".

- Use "block list" instead of "blacklist".

- Prefer user-focused language over system-focused language.

## Grammar

### Active / Passive Voice

- Prefer active voice.

### Pronouns

- Address users directly when appropriate.

### Sentence Length

- Prefer short, clear sentences.
## Lists

### Bulleted Lists

- Use bulleted lists for non-sequential information.
- Keep list items concise.
- Use parallel structure in list items.
- Maintain consistent wording and grammar.

### Numbered Lists

- Use numbered lists only for sequential steps.
- Present steps in the order users perform them.
- Keep instructions clear and action-oriented.

## Procedures

### Task Writing

- Focus on helping users complete tasks.
- Write from the user's perspective.
- Explain outcomes and benefits when helpful.

### Step Writing

- Begin steps with action verbs.
- Use clear and direct language.
- Do not include unnecessary actions.
- Do not assume information not provided.

## User Interface Language

### Buttons

- Use exact UI labels when available.
- Do not invent button names.

### Menus

- Use exact menu names when available.
- Do not invent navigation paths.

### Dialogs

- Use exact dialog names when available.
- Do not invent dialog titles.

### Messages

- Use clear, concise language.
- Avoid technical wording when simpler language works.

## Accessibility

### Inclusive Language

- Use inclusive and respectful language.
- Avoid biased or exclusionary terms.

### Bias-Free Language

- Prefer allow list over whitelist.
- Prefer block list over blacklist.

### Alt Text

- Provide meaningful descriptions when images are referenced.
- Focus on purpose, not visual appearance alone.

## Formatting

### Notes

- Use notes only for helpful supplementary information.

### Warnings

- Use warnings only for important risks or consequences.

### Code Examples

- Keep examples concise and relevant.
- Explain code when necessary.

### Links

- Use descriptive link text.
- Avoid vague references such as "click here".

## Examples

### Preferred Writing

Good:
Users can create a forecast scenario variant.

Bad:
The forecast scenario variant can be created by users.

---

Good:
Select Save.

Bad:
Please click on the Save button.

---

Good:
Sign in to your account.

Bad:
Login to your account.

---

Good:
Use the dropdown menu.

Bad:
Use the drop-down menu.

## Exceptions

- Use exact product terminology when required by the product.
- Use exact UI labels even if they differ from preferred terminology.
- Do not modify official product names.
- Do not change terminology defined by the customer style guide.
- Customer-specific terminology takes precedence over built-in terminology.
