# Microsoft Writing Style Guide

## Purpose
The guide provides writing style and terminology guidelines for all forms of communication related to computer technology, ensuring consistency and clarity.

## Audience
The guide is intended for anyone writing about technology, including app developers, website creators, and white paper authors.

## Tone
The tone is warm, relaxed, crisp, and clear, aiming to be approachable and straightforward.

## Voice
Microsoft's voice is simple and human, focusing on being warm, relaxed, crisp, clear, and ready to assist.

## Writing Principles
- Prioritize clarity and simplicity.
- Use conversational language.
- Avoid jargon and acronyms.
- Be concise and direct.
- Use contractions for a friendly tone.

## Headings

### Heading Rules
- Use sentence-style capitalization.
- Avoid end punctuation for titles, headings, and subheadings.

### Capitalization Rules
- Capitalize only the first word and proper nouns in headings.
- Use title-style capitalization for product names and formal titles.

## Terminology

### Preferred Terms
- Use "allow list" and "block list" instead of "whitelist" and "blacklist."
- Use "choose" instead of "select" when UI text begins with "Select."

### Avoided Terms
- Avoid "master/slave" terminology.
- Avoid "whitelist" and "blacklist."

### Terminology Examples
- Use "allow list" instead of "whitelist."
- Use "block list" instead of "blacklist."

## Grammar

### Active / Passive Voice
- Use active voice whenever possible.

### Pronouns
- Use second person (you) frequently.
- Avoid gender-specific pronouns; use "they" or role-based references.

### Sentence Length
- Favor short sentences for readability and scanning.

## Lists

### Bulleted Lists
- Use for items that do not require a specific order.

### Numbered Lists
- Use for step-by-step instructions or when order is important.

## Procedures

### Task Writing
- Use imperative verbs.
- Keep instructions concise and clear.

### Step Writing
- Use numbered steps.
- Write complete sentences for each step.

## User Interface Language

### Buttons
- Use verbs like "Select" or "Enter" for actions.

### Menus
- Use "Go to" for navigating menus.

### Dialogs
- Use "Open" and "Close" for dialogs.

### Messages
- Use clear, direct language for error and informational messages.

## Accessibility

### Inclusive Language
- Use people-first language where appropriate.
- Avoid terms like "special needs."

### Bias-Free Language
- Avoid gender-specific language and terms with negative connotations.

### Alt Text
- Provide descriptive alt text for images to ensure accessibility.

## Formatting

### Notes
- Use bold for emphasis, not all caps.

### Warnings
- Clearly distinguish warnings from regular text.

### Code Examples
- Use consistent formatting for code snippets.

### Links
- Use descriptive link text rather than "click here."

## Examples
- Use real-world examples to illustrate guidelines.

## Exceptions
- Use title-style capitalization in tweets for quoted articles.
- Use alternative input methods documentation when necessary.
